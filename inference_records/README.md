# 训练数据端到端推理记录（2026-07-19）

- 服务器：`A800_1`
- 服务目录：`/workspace/WM/dit4dit/DiT4DiT`
- checkpoint：`steps_48000_pytorch_model.pt`
- 输入：`pick_up_pipette_right_joints` 的 episode 0、frame 0
- 图像：RGB `224 x 224 x 3`
- 指令：`pick up the pipette`
- 状态：右臂 7 维 + Inspire 右手 6 维，共 13 维
- 输出：未来 16 步、每步 13 维，即 `16 x 13`

## 结果

服务器本机 ZMQ 请求成功，延迟 `1.1912 s`。第一步右臂预测为：

```text
[0.036363, -0.258599, -0.155906, -0.756981, 0.382491, 0.338458, -0.206456]
```

本机通过 SSH 隧道访问 A800 ZMQ 服务成功，端到端延迟 `1.9134 s`。第一步右臂预测为：

```text
[-0.023902, -0.256595, -0.103511, -0.724729, 0.372194, 0.350801, -0.254414]
```

两次结果不完全一致是因为动作扩散采样当前没有固定随机种子；两次结果均位于训练统计和
G1 URDF 关节限位内。完整 16 步结果见同目录两个 JSON 文件，复现输入保存在 NPZ 中。

测试过程中修复了三个会阻止实机推理的问题：

1. action-only checkpoint 不能按完整模型 strict load；现在由 Cosmos 基座加 action checkpoint 组合加载。
2. 状态 padding 原先硬编码为 64；现在从 checkpoint 的 state encoder 自动读取 16。
3. 真机客户端原先只发送 7 维右臂状态；现在追加 6 维右手状态，形成训练一致的 13 维输入。
